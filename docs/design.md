# 設計メモ（family-electricity-cost-share）

このドキュメントは、家族電気料金シェアアプリの設計メモです。  
まずはラフに書き出して、あとで必要に応じて更新していく。

---

※画面一覧・画面遷移図は画像で管理する。

- [ ] ここに「画面一覧」の画像を貼る（またはリンクを書く）
- [ ] ここに「画面遷移図」の画像を貼る（またはリンクを書く）

### メモ（文章で補足したいことがあれば）

- 例）ログイン後は◯◯画面に遷移する  
- 例）管理者だけが遷移できる画面 など

## ③ URL とルーティング

※ルーティング（URLとビュー関数（画面用の関数）の対応）の一覧を書く。

- [ ] ログイン関連のURL
- [ ] 使用セッション一覧・登録・編集・削除のURL
- [ ] 冷暖房器具の一覧・登録・編集のURL（管理者用）

  - 「自分の使用履歴を一覧表示する」など

## ④ ロールと権限の設計

※ロール（role＝ユーザーの役割）と、それぞれが何をできるかを書く。

### ロールの種類（例）

- [ ] 一般ユーザー（自分用のロール）
- [ ] 管理者ユーザー（家族代表）

### それぞれに許可する操作

- 一般ユーザー：
  - [ ] できること（例：自分のセッションのCRUD（基本操作））
  - [ ] できないこと（例：器具の登録・他人のデータ編集 など）
- 管理者：
  - [ ] できること（例：器具の登録・編集、全体の設定 など）
  - [ ] 将来やりたいこと（例：ユーザー管理 など）

---

## ⑤ データの流れ（入力 → DB → 画面）

※「ユーザーの操作」がどんな流れでDBや画面に反映されるかを書く。

### 例として書くと良さそうな流れ

- [ ] 使用セッションを新規登録するときの流れ
  - どの画面 → どのフォーム（入力欄） → どの処理 → どのテーブルに保存 → どの画面に戻る？
- [ ] 一覧画面でフィルタするときの流れ
  - どんなパラメータ（期間・ユーザーなど） → どんな条件でDBから取るか？
- [ ] 器具マスタ（冷暖房器具）を登録・編集するときの流れ（管理者）

ここは「箇条書きのストーリー」になっていればOK。  
あとでコードを書くときの道しるべにする。

## ⑥ バリデーション方針（入力チェック）

※バリデーション（validation＝入力が正しいかチェックすること）の方針を書く。

### 共通の方針

- [ ] 日付・時刻の必須チェック
- [ ] 数値（料金など）の範囲（マイナスを許さない など）
- [ ] フォームのエラー時にどう表示するか（メッセージの簡単なイメージ）

### 画面ごとの方針メモ

- 使用セッション登録・編集：
  - [ ] 開始時刻と終了時刻の関係（開始 ＜ 終了 など）
  - [ ] 器具が選ばれているか
- 器具登録・編集：
  - [ ] 名称の必須チェック
  - [ ] 1時間あたり料金のバリデーション

---

## ⑦ MVP範囲（今回の完成定義）

※MVP（Minimum Viable Product＝最小限の完成形）の定義を書く。

### 「ここまでできたら今回の提出として完成」とみなす条件

- [ ] ログイン／ログアウトができる
- [ ] 自分の冷暖房使用セッションを登録・編集・削除できる
- [ ] 一覧画面で自分の履歴が見える
- [ ] 料金が自動計算されて保存される
- [ ] 一覧画面で、期間絞り込み＋合計金額の表示ができる
- [ ] READMEに、目的と機能概要とMVPの説明が書いてある

### 「今回はやらない」と決めたこと（将来やりたいこと）

- [ ] 家族全員の最終負担額の自動計算
- [ ] 締め日ロジック
- [ ] グラフ表示
- [ ] ダークモード
- [ ] 通知機能 など

ここに「やらない」と決めた理由も一言書いておくと、  
面接やプレゼンで説明しやすくなる。
